# WO Lifecycle / Harness Anchor

## Purpose

This document is the **single source of truth for the WO/lifecycle/harness program roadmap**.
It defines the intended migration path, stable program decisions, active phase ordering, and closure criteria for the "WO sin fricción" initiative.

This anchor is **planning authority only**.
It is **not** authority over:

- the live runtime behavior,
- the current state of any specific WO,
- transactional state under `_ctx/jobs/**`, or
- operational evidence generated by the scripts.

Runtime authority remains in the official lifecycle scripts and the state they manage. Tactical fixes, local cleanups, and migration batches must align to this anchor rather than silently redefining the roadmap.

> **Durability note:** This anchor only counts as the SSOT of the program once it is contained in a durable repository reference (for example, a tracked branch or merged repo history). A copy that exists only inside a local worktree is still a draft carrier, not durable program authority.

---

## Final Objective

A **WO sin fricción** exists when the lifecycle is operationally boring, deterministic, and auditable:

1. an operator or agent uses one official path to take, verify, and close a WO;
2. each lifecycle phase has a single authority surface;
3. evidence is produced reliably without being treated as state authority;
4. closure outcomes are repeatable from a clean worktree without manual repair;
5. rescue flows exist, but they are not part of the happy path;
6. a closeout can be audited without reconstructing undocumented side effects.

This objective is considered reached only when those properties are demonstrable in docs, harness behavior, and contract tests.

---

## Non-Negotiable Principles

1. **One authority per phase**  
   Every lifecycle phase must have exactly one owner surface for decisions and mutation.

2. **Evidence != authority**  
   `verdict.json`, logs, patches, and reports are evidence outputs; they do not own lifecycle state transitions by themselves.

3. **No manual state surgery**  
   Humans and ad-hoc scripts must not move YAMLs, rewrite state by hand, or simulate lifecycle transitions outside official entrypoints.

4. **No multiple sources of truth**  
   Docs, wrappers, helpers, and tests may explain the flow, but they must not redefine ownership.

5. **Fail-closed, but scoped**  
   Gates should block on violations relevant to their authority boundary, without widening into unrelated repo debt.

6. **Worktree as managed resource**  
   Worktrees are lifecycle resources created, inspected, and cleaned through managed flows, not disposable side effects.

7. **Rescue surfaces stay outside the happy path**  
   Reconcile, GC, and repair tools exist for interruption recovery, not as normal substitutes for take/verify/finish.

## Certified Current State

Only the following outcomes are treated as currently consolidated for the program:

- **Crash-safety of `finish` improved in certified worked scenarios**  
  Closeout hardening work certified concrete crash-safety and post-failure recovery expectations in the finish scenarios already exercised; it does not yet certify the entire subsystem as globally complete.

- **Scoped closeout authority established**  
  WO closure is no longer treated as an open-ended repo-wide action; the closeout path has a WO-scoped authority model and explicit finish-time checks.

- **`state_root` / `execution_root` separation established**  
  The lifecycle now distinguishes shared WO state location from the execution cwd/worktree context, reducing topology-dependent behavior.

- **Official `failed -> pending` path restored**  
  Requeue exists again as an official lifecycle operation instead of requiring informal repair behavior.

- **Ownership-aware gates proven in worked cases**  
  In the cases already remediated, gate boundaries were tightened toward ownership-aware behavior instead of treating every verification surface as one global authority.

These items are certified as program progress, but they do **not** imply the full WO/lifecycle/harness program is complete.

## Irreversible Decisions Already Taken

The following program decisions are locked unless superseded by an explicit ADR that references this anchor:

- **`ctx_wo_finish.py` is the only terminalizer / state-owner** for WO closeout.
- **`wo_verify.sh` is evidence/verdict-owner only** and must not be treated as the lifecycle state authority.
- **`verify.sh` and `repo_verify.sh` are outside the WO lifecycle**; they are global/repo hygiene surfaces, not WO terminalizers.
- **`ctx_verify_wo.py` is deprecated** and may remain only as a compatibility surface during migration.
- **`gc` and `reconcile` are rescue-only** and must remain outside the happy-path lifecycle.

## Active Phase Now: V3 Migration

The active phase is the **V3 contractual migration**. The safe order is mandatory:

1. **Publish ADR/spec**  
   Freeze the V3 ownership model in an explicit architecture/spec artifact.

2. **Align docs/help**  
   Update normative docs and CLI/help text so the official lifecycle story matches the target harness.

3. **Add contract tests**  
   Lock ownership boundaries before changing runtime semantics.

4. **Migrate references**  
   Update templates, examples, defaults, and legacy references that still point operators toward deprecated surfaces.

5. **Activate blocks**  
   Turn legacy references into CI/test failures first, then tighten runtime/operator guidance.

6. **Only then change `wo_verify.sh` semantics**  
   Remove any remaining implicit lifecycle-state ownership from `wo_verify.sh` only after the harness, docs, and tests are ready to absorb it safely.

This order exists to avoid a broken migration in which `wo_verify.sh` stops mutating state before the harness and contract surfaces are prepared.

---

## Residual Risks

### Primary Transition Risk

- **`verdict_but_running`**  
  The most likely migration failure is producing PASS/FAIL verdict evidence while the WO remains in `running/` because state terminalization was not completed by the official harness.

### Active Watchpoints

- **Internal side effects around closeout orchestration**  
  Any internal helper or extracted closeout routine (including a future `execute_closeout_action(...)`-style helper) must be treated as a watchpoint: side effects must remain explicit, testable, and subordinate to the single terminalizer contract.

- **Legacy references acting like authority**  
  Deprecated docs/help/examples can silently reintroduce multiple truths even when runtime code is partially correct.

- **Wrapper drift**  
  Compatibility wrappers are acceptable only if they preserve the ownership model and do not present themselves as competing lifecycle authorities.

## Anchor Governance Rule

This anchor governs the program roadmap.

- Every new WO/lifecycle/harness batch must **reference this anchor**.
- Every new ADR affecting WO lifecycle ownership must **link back to this anchor**.
- Every checkpoint/handoff must state explicitly whether it **changes** or **does not change** the anchor.
- Tactical checkpoints, local remediation notes, and batch plans **do not redefine the roadmap** by themselves.
- If a checkpoint contradicts this anchor without an explicit superseding ADR, the checkpoint is treated as tactical context, not roadmap authority.
